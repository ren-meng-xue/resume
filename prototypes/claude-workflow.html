// 全局约束
项目约束（始终生效，任何情况不可跳过）
违反视为严重问题
#1  语言
始终使用简体中文
所有回复始终使用简体中文，任何情况不可跳过
#2  安全配置
禁止硬编码 Key / Secret
所有配置走环境变量，禁止在代码中硬编码任何 Key 或 Secret
#3  commit / push
用户指令直接执行；Claude 主动发起须等「1」
用户消息本身即指令 → 直接执行
Claude 主动提交 → 说明内容，等用户回复「1」
push 后非 main → 询问是否合并
#4  LLM 调用层级
services/ 层唯一调用点
LLM 调用只在 services/ 层，api/ 层禁止直接调用
#5  实时状态
SSE + Redis Pub/Sub
实时状态用 SSE + Redis Pub/Sub，禁止轮询
#6  数据库变更
必须走 Alembic
数据库变更必须走 Alembic 迁移，禁止直接修改表结构。ORM 模型变更后迁移步骤见 deploy.md
#7  开发环境
./dev.sh 一键启动（混合模式）
Docker 跑 postgres + redis，其余服务直接在本机跑。新增或删除服务时必须同步更新 dev.sh
#8  会话机制
分级触发 + Changelog
会话开始 / 任务前采用分级触发机制（L0 必做 → L1/L2 按需），changelog 读写与更新规则见 changelog.md
#9  分支切换
切换前必须 git status
切换分支前必须先执行 git status 检查未提交改动。如有，列出清单询问用户：先 commit / stash 再切，还是直接切
#10  开发顺序
先实现，后测试（非 TDD）
先写实现代码，实现完成后再写测试并运行通过。不使用「先写失败测试」的 TDD 流程
#11  计划前分支检查
制定计划 / 使用 writing-plans 前，必须执行以下 6 步
a. 执行 git branch --show-current 获取当前分支名
b. 判断计划内容是否为独立新功能（与当前分支职责无关）
c. 若属新功能：询问「是否创建新分支？建议名称：feature/xxx」
d. 执行 git status：有改动时列出清单并询问 commit / stash / 留当前分支
e. 等待用户回应（语义判断：「ok」「1」「好」均视为同意）再开始制定计划
f. 若当前在 main 且计划需要独立分支，必须提醒，不可直接在 main 开发
